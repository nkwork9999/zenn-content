# アイデア

化学データ系のコンペの[追試](https://zenn.dev/nkwork9999/articles/neurips-polymer-2025)でヒントを得て、化学データを扱える自作拡張機能を作り始めた。

### できること
- pandasやRDKitなしで、SMILES文字列やInChI、PDB...など化学データをそのまま扱える
- DuckDB標準機能でCSV/Parquet/Iceberg/S3/HTTP直URLとそのまま繋がるので、データ取り込みから軽微な前処理までできる


# そもそも

化学データって？というと、分子の構造を「文字」で表したSMILESという記法がある。

このデータを読み取ったり前処理したりするライブラリとして有名なのがRDKitである。
例えばエタノールはSMILESでは`CCO`と記載されるが、RDKitで分子式を出力すると`C2H6O`という様になる。

それだけではなく、分子量やフィンガープリント、分子記述子、部分構造検索...など非常に多岐にわたる処理ができる。おそらく化学系のデータ処理には必須のライブラリなのではないかと思う。(中身はさまざまな論文に基づいたアルゴリズムが...)

じゃああえて拡張機能を作る必要はない気もするが、使用した感じでは以下のような問題があった。

・google colab上で動かなかったりする(ランタイム再起動が必要)
・pandasのバージョン関連の競合が起こる
・データを出力する場合にはpandasを結局経由したり、機能によっては別ツールとの連携が必要になる


# 作ったもの

ということでduckdbの拡張機能として以下を作ってみた。

## ducksmiles
https://duckdb.org/community_extensions/extensions/ducksmiles

RDKitでできることや、RDKitだけだと面倒なことを混ぜつつ、いくつかの機能を実装してみた。

1. SMILES文字列から分子式・分子量を計算してCSVに書き戻す
   → 化学データの一番基本の処理をSQL1行で完結させる
2. InChIという別の表記を、結合・水素・電荷・立体の各層に分解して取り出す。骨格レベルでの一致比較もできる
   → RDKitだとInChIを一度分子に変換し直さないと触れない情報を、文字列のまま分解できる
3. SMILESとSELFIES（機械学習で扱いやすい別表記）の相互変換
   → RDKitでは別パッケージが必要な処理をDuckDB単体で完結させる
4. PDB/CIF/XYZ（タンパク質などの3D構造ファイル）から原子数・鎖数・残基数をSQLで集計
   → 重い構造ファイルでも、開かずにSQLだけでメタ情報を取れる
5. 不正なSMILESや構造データが混じってもエラーを出さずNULLで返す
   → 数百万件のバッチ処理で1件のミスで全部止まるのを防ぐ


# 使ってみる

公開データ（[DeepChem ESOL CSV](https://deepchemdata.s3-us-west-1.amazonaws.com/datasets/delaney-processed.csv) と [RCSB PDB 1AKE](https://files.rcsb.org/download/1AKE.pdb)）を使って、できること1〜5を実際にCLIで動かしたログを貼っていく。

```sh
duckdb
```

```sql
INSTALL ducksmiles FROM community;
LOAD ducksmiles;
```

## 1. SMILES → 分子式・分子量

```sql
SELECT smiles, mol_formula(smiles) AS formula, mol_weight(smiles) AS weight
FROM read_csv_auto('https://deepchemdata.s3-us-west-1.amazonaws.com/datasets/delaney-processed.csv')
LIMIT 5;
```

```
┌────────────────────────────────────────────────────────┬───────────┬─────────┐
│                         smiles                         │  formula  │ weight  │
├────────────────────────────────────────────────────────┼───────────┼─────────┤
│ OCC3OC(OCC2OC(OC(C#N)c1ccccc1)C(O)C(O)C2O)C(O)C(O)C3O  │ NULL      │ NULL    │
│ Cc1occc1C(=O)Nc2ccccc2                                 │ C12H11NO2 │ 201.225 │
│ CC(C)=CCCC(C)=CC(=O)                                   │ C10H16O   │ 152.237 │
│ c1ccc2c(c1)ccc3c2ccc4c5ccccc5ccc43                     │ C22H14    │ 278.354 │
│ c1ccsc1                                                │ C4H4S     │ 84.136  │
└────────────────────────────────────────────────────────┴───────────┴─────────┘
```

公開URLを直接`read_csv_auto`に渡して、その場で分子式と分子量が出る。1行目のように複雑なSMILESではNULLが返るが、例外にはならない。

## 2. InChI の各層を分解する

```sql
SELECT
  name,
  inchi_formula(inchi) AS formula,
  inchi_connections(inchi) AS connections,
  inchi_hydrogens(inchi) AS hydrogens
FROM (VALUES
  ('Ethanol',  'InChI=1S/C2H6O/c1-2-3/h3H,2H2,1H3'),
  ('Caffeine', 'InChI=1S/C8H10N4O2/c1-10-4-9-6-5(10)7(13)12(3)8(14)11(6)2/h4H,1-3H3')
) AS t(name, inchi);
```

```
┌──────────┬───────────┬───────────────────────────────────────┬────────────┐
│   name   │  formula  │              connections              │ hydrogens  │
├──────────┼───────────┼───────────────────────────────────────┼────────────┤
│ Ethanol  │ C2H6O     │ 1-2-3                                 │ 3H,2H2,1H3 │
│ Caffeine │ C8H10N4O2 │ 1-10-4-9-6-5(10)7(13)12(3)8(14)11(6)2 │ 4H,1-3H3   │
└──────────┴───────────┴───────────────────────────────────────┴────────────┘
```

InChIはもともと原子のつながり方や水素の位置といった情報を、別々のパートに分けて1本の文字列にまとめた表記。例えばエタノールはこう書ける：

```
InChI=1S/C2H6O/c1-2-3/h3H,2H2,1H3
        └─────┘ └────┘ └────────┘
        分子式  結合     水素の位置
       inchi_formula  inchi_connections  inchi_hydrogens
```

普通はRDKitで一度分子に戻してから情報を取り出すが、ducksmilesなら欲しいパートだけを文字列のまま直接取り出せる。実際、上のクエリ結果（`C2H6O` / `1-2-3` / `3H,2H2,1H3`）は文字列中の対応するパートをそのまま抜いたものになっている。

## 3. SMILES ⇄ SELFIES の相互変換

ESOLの中から短めのSMILESを引っ張ってきて、SMILES → SELFIES → SMILES の往復をかけてみる。

```sql
SELECT
  smiles,
  smiles_to_selfies(smiles) AS selfies,
  selfies_to_smiles(smiles_to_selfies(smiles)) AS roundtrip
FROM read_csv_auto('https://deepchemdata.s3-us-west-1.amazonaws.com/datasets/delaney-processed.csv')
WHERE LENGTH(smiles) < 12
LIMIT 5;
```

```
┌───────────┬─────────────────────────────────────────┬───────────┐
│  smiles   │                 selfies                 │ roundtrip │
├───────────┼─────────────────────────────────────────┼───────────┤
│ c1ccsc1   │ [c][Ring1][C][c][c][s][c][Ring1][C]     │ cccsc     │
│ O=C1CCCN1 │ [O][=C][Ring1][C][C][C][C][N][Ring1][C] │ O=CCCCN   │
│ CCCC=C    │ [C][C][C][C][=C]                        │ CCCC=C    │
│ CC(C)Cl   │ [C][C][Branch1][C][C][Cl]               │ CC(C)Cl   │
│ CCC(C)CO  │ [C][C][C][Branch1][C][C][C][O]          │ CCC(C)CO  │
└───────────┴─────────────────────────────────────────┴───────────┘
```

DuckDBの中だけで両方向の変換が回る。RDKitだと別パッケージ(`selfies`)を入れる必要がある処理。

## 4. PDB構造ファイルのメタ情報

`read_text`はHTTPS URLや配列をそのまま受け付けるので、RCSB PDBから直接3ファイル分まとめて読み込める。

```sql
INSTALL httpfs; LOAD httpfs;

SELECT
  filename,
  structure_atom_count(content)    AS atoms,
  structure_chain_count(content)   AS chains,
  structure_residue_count(content) AS residues,
  structure_model_count(content)   AS models
FROM read_text([
  'https://files.rcsb.org/download/1AKE.pdb',
  'https://files.rcsb.org/download/1CRN.pdb',
  'https://files.rcsb.org/download/4HHB.pdb'
]);
```

```
┌──────────────────────────────────────────┬───────┬────────┬──────────┬────────┐
│                 filename                 │ atoms │ chains │ residues │ models │
├──────────────────────────────────────────┼───────┼────────┼──────────┼────────┤
│ https://files.rcsb.org/download/1AKE.pdb │ 3816  │ 2      │ 428      │ 1      │
│ https://files.rcsb.org/download/1CRN.pdb │ 327   │ 1      │ 46       │ 1      │
│ https://files.rcsb.org/download/4HHB.pdb │ 4779  │ 4      │ 574      │ 1      │
└──────────────────────────────────────────┴───────┴────────┴──────────┴────────┘
```

`read_text`が中身を`content`列として返してくれるので、そのまま`structure_*`関数に渡すだけ。`filename`列も勝手に付いてくるので、数千ファイル単位で集計しても元ファイルが追いやすい。

## 5. 不正データでもエラーで止まらず NULL

```sql
SELECT
  smiles,
  mol_is_valid(smiles) AS valid,
  mol_formula(smiles) AS formula,
  mol_weight(smiles) AS weight
FROM (VALUES
  ('CCO'),
  ('not_a_smiles'),
  ('c1ccccc1'),
  (''),
  ('CC(=O)O')
) AS t(smiles);
```

```
┌──────────────┬───────┬─────────┬────────┐
│    smiles    │ valid │ formula │ weight │
├──────────────┼───────┼─────────┼────────┤
│ CCO          │ true  │ C2H6O   │ 46.069 │
│ not_a_smiles │ false │ NULL    │ NULL   │
│ c1ccccc1     │ true  │ C6H6    │ 78.114 │
│              │ false │ NULL    │ NULL   │
│ CC(=O)O      │ true  │ C2H4O2  │ 60.052 │
└──────────────┴───────┴─────────┴────────┘
```

不正なSMILESや空文字が混じっても、行ごとにNULLになるだけでクエリは止まらない。バッチ処理向きの作りになっている。


# 雑感

化学系のデータを扱ってて思ったのは、とにかくフォーマットが多いということ。  
SMILES、InChI、SDF...と固有のデータ形式があり、それぞれに別途専用のツールが必要になる。  

他にもRDKitは非常に多くの機能を持っていて、「記述子」だけでも200種類くらいあり、それぞれが色々な論文を参照している。

ducksmilesは今後はこうした拡げがいのある領域を、先人の知恵を読み解きつつ、パイプ役・前処理を、なんとかツール一個で速くできるようなものにしていけたらと思っている。

また、DuckDB本体は汎用エンジンなので、各ドメインの細かい要求は拾いきれない。  
そこを埋めるのがコミュニティエクステンションの役割で、どんどん個人でも作れるので挑戦してみて欲しい。
