# Chapter 1: Tensors and Shapes

`einsum` 記法に沿って tensor 操作する手法を深ぼる。

## 1. Einstein notation でテンソル操作を覚える

dmol本文の練習。rank 3のテンソル `a` (shape (4,3,2)) を作って、軸ごとの集約を試す。

単純なリシェイプ
```python
a = np.arange(24).reshape(4, 3, 2)
```

```python
b = np.sum(a, axis=0)        # axis=0 を潰す
c = np.any(a > 4, axis=1)    # axis=1 を潰す、bool化
d = np.prod(a, axis=(2, 1))  # axis=2 と axis=1 を両方潰す
```


### `np.sum` と `np.einsum` の対応

さっきの `np.sum(a, axis=0)` を einsum で書くと:

```python
b1 = np.sum(a, axis=0)
b2 = np.einsum('ijk->jk', a)
print("一致:", np.array_equal(b1, b2))
# 一致: True
```

`'ijk->jk'` の読み方:
- 入力 `a` の3軸を、それぞれ `i, j, k` で名付ける
- `->` の右が出力の形 (`jk`)
- 左にあって右にない添字 (`i`) は「和を取って消す」 = `axis=0` を潰すのと同じ

※和をとって消す
指定した軸の方向に全要素を足し合わせる
その軸はもう必要ないので、shape から削除される

### einsum の文法を1行で

| 書き方 | 意味 (numpy的に) |
|---|---|
| `'ij->i'` | 行列の行ごとの和 (`M.sum(axis=1)`) |
| `'ij->'` | 全要素の和 (`M.sum()`) |
| `'ij,j->i'` | 行列ベクトル積 (`M @ v`) |
| `'ij,jk->ik'` | 行列の積 (`A @ B`) |
| `'bij,bjk->bik'` | バッチ単位の行列積 (`b` を揃えて、各バッチで `A @ B`) |


## 2. Broadcasting

ひとことで言うと: 「形 (shape) が違うテンソル同士でも、numpy が自動で形を合わせて演算してくれる仕組み」。

### Step 1: 一番シンプルな例 (スカラー + 配列)

数の並び (配列) に1つの数を足すと、全要素に同じ数が足される。これが broadcasting基本

```
[1, 2, 3] + 10  →  [11, 12, 13]
```

「10」というスカラーが、配列の各要素に自動で足された。forループを書家内で良い。numpy が裏でやっている。

### Step 2: 行列 + ベクトル

例えば「分子5個 × 特徴3個」の表があって、各列の平均を引きたいとする (= 中央化):

```
                   特徴1   特徴2   特徴3
分子0           [   5.1,    3.0,   12.0 ]
分子1           [   5.0,    3.1,   12.1 ]
分子2           [   4.9,    2.9,   11.9 ]
分子3           [   5.2,    3.2,   12.2 ]
分子4           [   4.8,    2.8,   11.8 ]   ← 表の形は (5, 3)

平均ベクトル    [   5.0,    3.0,   12.0 ]   ← 形は (3,)
```

形が違う `(5, 3)` と `(3,)` を引き算するのは、本来なら無理に見える。でも numpy は「平均ベクトルを5回コピーして縦に5行に並べた仮想的な (5, 3) 行列」として扱い、引き算してくれる:

```python
中央化された表 = 表 - 平均ベクトル
# (5, 3) - (3,) → 自動で (5, 3) - (5, 3) として処理 → (5, 3)
```

引き算結果:

```
                   特徴1   特徴2   特徴3
分子0           [  +0.1,    0.0,    0.0 ]
分子1           [   0.0,   +0.1,   +0.1 ]
分子2           [  -0.1,   -0.1,   -0.1 ]
分子3           [  +0.2,   +0.2,   +0.2 ]
分子4           [  -0.2,   -0.2,   -0.2 ]   ← 各列が平均0を中心に分布
```

各列ごとに**異なる平均値** (5.0 / 3.0 / 12.0) が、自動でそれぞれの列に引かれている。これが broadcasting。

### Step 3: なぜ嬉しいか

- forループより何十〜何百倍も速く、短く書ける。
- ML / データ前処理は「全行に同じ計算をする」が多いため。

### Step 4: 化学データでの典型例 (3次元テンソル)


| 種類 | 形 | 意味 |
|---|---|---|
| 入力 | `(1000分子, 50原子, 3座標)` | 1000分子の3D構造 |
| 重心 | `(1000分子, 3座標)` | 各分子で原子方向に平均した値 |
| ほしい結果 | `(1000分子, 50原子, 3座標)` | 各原子から自分の分子の重心を引いたもの |

重心 = その分子の全原子の座標の中心点。例えば3原子の分子の各座標が `(0,0,0)`, `(2,0,0)`, `(1,3,0)` なら、重心は平均をとって `(1,1,0)`。
なぜ重心を引くか
<br>
同じ分子でも置き場所によって絶対座標は変わるが、重心を原点 `(0,0,0)` に揃えると「**形そのもの**」だけが残る。「分子を平行移動しても同じ分子として扱う」ためにほぼ必須の前処理。

## 3. Modifying rank — 「次元の数」を増やす・減らす・並べ替える

テンソルが持つ「軸の数」を **rank (ランク)** と呼ぶ。例えば:
- スカラー (1個の数値) は rank 0
- ベクトル (1次元配列) は rank 1
- 行列 (2次元配列) は rank 2
- 3Dテンソル (3次元配列) は rank 3

化学データで実際に出てくる形を整理しておくとイメージしやすい:

| 形 | 意味 |
|---|---|
| `(N分子,)` | 各分子のラベル (溶解度など) |
| `(N分子, D特徴)` | 分子の記述子表 (今回の AqSolDB の17次元など) |
| `(N分子, A原子, 3)` | 各分子の3D座標 (xyz) |
| `(N分子, A原子, F)` | 各原子の特徴量 (GNN入力) |
| `(N分子, A原子, A原子)` | 距離行列・隣接行列 |

### なぜ rank を変えたいか

実用で rank を変える理由はだいたいこの3つ:

1. **broadcasting を成立させたい**
   - `(9982, 17)` の表から `(9982,)` のラベルを引きたい → ラベルを `(9982, 1)` に直す必要がある (前章で見たやつ)
2. **ライブラリ・レイヤが期待する形に合わせたい**
   - モデルが「バッチ形式 `(N, ...)`」を期待 → 単一分子でも `(1, 50, 3)` のように1次元増やす必要
   - 線形層 (Linear/MLP) は2D入力 `(N, D)` を期待 → 3D構造 `(N, A, 3)` を `(N, A*3)` に潰す必要
3. **不要な軸を消してスッキリさせたい**
   - 推論結果が `(1, 1)` で返ってきた → スカラー `()` にしたい



### `newaxis`: 「長さ1の軸」を新しく入れる

例: 各分子の溶解度ラベル `(9982,)` を、記述子表 `(9982, 17)` と broadcast 計算したい場合。

```
labels                  : (9982,)
labels[:, np.newaxis]   : (9982, 1)    ← 2次元化、broadcast可能に
```

これで `features - labels[:, np.newaxis]` のように、各分子の17特徴量からその分子のラベルを引く処理が成立する。

別の例: 各原子の部分電荷 `(50原子,)` を、その原子のxyz座標 `(50原子, 3)` と並べて「電荷+xyz」の4次元特徴ベクトルにしたい:

```
charges                 : (50,)
xyz                     : (50, 3)
charges[:, np.newaxis]  : (50, 1)
concatenate([xyz, charges[:, np.newaxis]], axis=1) → (50, 4)
```

### `squeeze`: 「長さ1の軸」を全部削除

化学で典型的なのは「**単一分子をモデルに渡したら、バッチ次元1がついて返ってきた**」ケース:

```
入力 (単一分子の3D構造): (1, 50, 3)    ← モデルがバッチ形式を期待するので (1, ...) で渡す
出力 (1分子のスカラー予測): (1, 1)
squeeze(出力) → ()                     ← 余分な軸を削って素のスカラーに
```

GNN/Transformer周りでバッチ次元1が紛れ込むのを掃除する場面が多い。

### `reshape`: 形を自由に並べ替え

要素の中身は変えずに、入れ物の形だけ変える操作。中身の総数 (要素数) が同じなら、好きな形にできる。

化学データでの典型: **GNNを通した後の全原子特徴を、分子の境界を取っ払って1つの大きな表にする**。

```
バッチ内の全分子の全原子特徴: (32分子, 50原子, 64特徴)
                              ↓ reshape(-1, 64)
全原子フラット表           : (1600原子, 64特徴)
```

別の例: 3D座標を線形層 (MLP) に入れたい時、`(N分子, A原子, 3)` を `(N分子, A原子×3)` に潰す:

```
xyz座標   : (1000, 50, 3)
            ↓ reshape(1000, -1)
flatベクトル: (1000, 150)        ← 50×3=150 を自動計算
```

各分子を「150次元の平らなベクトル」として表現でき、MLPやLinear層に直接渡せる。

### `-1` の便利機能

上の例で使った `reshape(-1, 64)` や `reshape(1000, -1)` のように、**1箇所だけ `-1` を指定すると、numpyが要素数から自動計算してくれる**。

```
(1000, 50, 3) → reshape(1000, -1) → (1000, 150)
                              ↑
                        自動で 50×3=150
```

「**分子数や原子数が変わってもコードを書き換えなくていい**」のが `-1` のありがたさ。実装を分子サイズに依存させない、汎用的な前処理コードが書ける。

### dmol筆者の好み

dmol本文の脇注に「`b[:, None]` のような `None` 代用は推奨しない、 `np.newaxis` の方を使え」と書かれている。理由は **「`newaxis` の方が意図が明示的に読める」** から。

## 4. Rank slicing — `...` (省略記号) で「rank が分からないテンソル」を扱う

スライスを書くとき、テンソルの軸の数 (rank) が分かっていないと困ることがある。例えば「**最後の軸の0番目だけ取りたい**」という処理:

```python
a[:, 0]          # rank 2 のとき
a[:, :, 0]       # rank 3 のとき
a[:, :, :, 0]    # rank 4 のとき
```

`:` の数を rank に合わせて変える必要があり、汎用的な関数が書けない。

そこで Python の **省略記号 `...` (ellipsis)** を使う:

```python
a[..., 0]    # rank がいくつでも、最後の軸の0番目を取る
```

「**前にある全部の軸はそのまま、最後の軸の0番目を取る**」という意味で、rankに関係なく同じコードが動く。

### なぜ嬉しいか

GNN や Transformer では、入力テンソルの形が状況で変わる:

- 単一サンプル: `(原子数, 特徴次元)` = rank 2
- バッチ処理: `(バッチ, 原子数, 特徴次元)` = rank 3
- 時系列バッチ: `(バッチ, 時刻, 原子数, 特徴次元)` = rank 4

これらに同じ関数を使い回したい。`...` を使った関数は、どの rank の入力でも自然に動く。

## 5. View vs Copy

「テンソルをスライスしたとき、新しいテンソルは元のメモリを共有しているか、独立にコピーされているか?」 という話。

### View = 「同じ物体を2つの名前で呼んでる」状態

例: 分子の3D座標 `(50原子, 3座標)` から、最初の10原子だけ取り出す。

```
xyz_full = ...                  # (50, 3) 全原子の座標
xyz_head = xyz_full[:10]        # (10, 3) 最初の10原子
```

このとき `xyz_head` は view (元の `xyz_full` のメモリを共有) になっている。つまり片方をいじるともう片方も変わる:

```
xyz_head[0] = [999, 999, 999]   # ← xyz_head の0番目を上書き
print(xyz_full[0])              # → [999, 999, 999]   ← xyz_full の0番目も変わってる!
```

「同じ物体を2つの名前 (`xyz_full` と `xyz_head`) で参照してるだけ」だから、片方の変更が両方に伝わる。

### Copy = 「独立した別物」

明示的に `.copy()` を呼ぶか、演算 (`+`, `*`, `-` など) を介すと、独立したテンソルになる:

```
xyz_head = xyz_full[:10].copy()    # ← 明示的にコピー
xyz_head[0] = [999, 999, 999]
print(xyz_full[0])                 # → 元の値のまま、影響なし
```

### なぜ気にする必要があるか — 典型バグ

化学データの前処理でうっかり書きがちな例:

```
xyz_subset = xyz_full[:10]          # view!
xyz_subset -= center                # ← xyz_full の最初の10原子も中央化されてしまう
                                    #   後で xyz_full を再利用すると壊れたデータが入ってる
```

意図せず元データが壊れて、原因不明のバグになる。

### dmol筆者の脇注: ML本体ではほぼ気にしなくていい

> 多くのMLフレームワーク (JAX, TensorFlow, PyTorch autograd) では **そもそも要素代入そのものができない** (`x[0] = 5` がエラーになる、autodiffと干渉するため)。だから「view か copy か」の区別は ML本体では実質的に発生しない。
